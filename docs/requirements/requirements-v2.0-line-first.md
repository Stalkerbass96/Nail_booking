# 需求冻结结论（V2.0 LINE First）

- 冻结日期：2026-03-08
- 适用范围：系统 2.0 升级
- 目标：把系统从“网页预约为主”升级为“LINE 导流预约为主”

## 1. 角色

- 顾客：通过 LINE 加好友后进入预约系统，浏览图墙、提交预约、查看预约详情。
- 店长：维护图墙、套餐、预约、顾客资料，并通过后台与顾客 1 对 1 文本沟通。

## 2. 核心场景

### 2.1 顾客预约场景

1. 顾客线下扫码，添加店铺 LINE 官方账号为好友。
2. 系统根据 `LINE userId` 自动创建顾客档案。
3. 初始顾客状态为“潜在客户（未预约）”。
4. LINE 首次主动向顾客发送预约系统首页链接。
5. 顾客打开首页，看到类似 Instagram 的图墙。
6. 图墙中的每个展示项包含图片、标题、价格文案，并关联后台套餐。
7. 顾客点击某个图墙项后进入预约流程。
8. 顾客只能按该图墙项绑定的套餐预约，不允许中途改套餐。
9. 顾客填写：到店时间、备注。
10. 顾客提交预约后：
   - 预约状态为 `pending`
   - 档期立即占用
   - 顾客状态从潜在客户转为正式顾客
   - LINE 回消息：等待确认中 + 预约号 + 预约详情链接
11. 店长后台确认预约后，LINE 回消息：预约已确认。

### 2.2 店长管理场景

1. 店长后台维护图墙展示项，并为展示项关联套餐。
2. 图墙支持分类筛选，但不改变图墙整体排版方式。
3. 多个图墙展示项可以关联同一个套餐。
4. 店长在后台维护顾客资料，默认姓名使用 LINE 昵称，后台可手动修改。
5. 店长可以在后台基于 LINE 会话与顾客进行 1 对 1 文本沟通，以确认更细需求。
6. 当前其他预约、排班、封锁、积分等逻辑可继续沿用。

## 3. 顾客身份与资料规则

1. 顾客主身份为 `LINE userId`。
2. 加好友后立即创建顾客档案。
3. 未预约过的顾客状态为 `lead`（潜在客户）。
4. 提交首笔预约后，顾客状态变为 `active`（正式顾客）。
5. 邮箱不再作为主识别字段。
6. 邮箱不是必填项，仅作为辅助信息保留。
7. 后台必须能看到“已加好友但尚未预约”的顾客。

## 4. 图墙规则

1. 首页就是图墙首页。
2. 图墙展示项至少包含：
   - 封面图
   - 标题
   - 简短说明
   - 分类
   - 价格展示文案
   - 关联套餐
   - 排序
   - 上下架状态
3. 每个图墙展示项仅使用单图，不做多图详情。
4. 图墙支持分类筛选。
5. 筛选不改变图墙的整体瀑布流/宫格排版结构。
6. 价格展示使用“套餐基础价，文案可写‘起’”。

## 5. 预约与消息规则

1. 预约状态继续保留：
   - `pending`
   - `confirmed`
   - `completed`
   - `canceled`
2. `pending` 继续占用档期。
3. 店长确认预约后，LINE 主动通知顾客。
4. 顾客提交预约后，LINE 消息必须包含：
   - 等待确认说明
   - 预约号
   - 预约详情公开链接
5. 预约详情公开链接保留，但核心入口仍从 LINE 进入。
6. 原“邮箱 + 预约号查询预约”路径不再保留为 2.0 主流程。

## 6. LINE 会话规则

1. 顾客加好友后只首次推送首页链接。
2. 不要求所有顾客都有 LINE，但 2.0 主流程以 LINE 顾客为中心。
3. 店长与顾客 1 对 1 会话阶段只支持文本消息。
4. 不做图片消息、模板消息、快捷短语等扩展能力。

## 7. 继续沿用的规则

1. 单店模式。
2. 单店长/单技师模型。
3. 不支持指定技师。
4. 排班、封锁、特殊营业日逻辑继续保留。
5. 积分逻辑继续保留。
6. 支付仍为线下处理。
7. 中日双语继续保留。

## 8. 关键冻结值

- 顾客主身份：`LINE userId`
- 首页入口：图墙首页 `/`
- 图墙与套餐关系：多个图墙项可指向同一套餐
- 套餐切换：不允许
- 顾客补充字段：不增加
- 顾客默认姓名：LINE 昵称
- 正式顾客转化时点：提交预约即转正式
- LINE 首页链接推送时机：仅首次加好友

## 9. 不在 2.0 首批范围

- 邮箱驱动预约查询
- 图墙多图详情
- 顾客预约时额外采集电话/姓名/邮箱
- 预约流程中的套餐切换
- LINE 图片消息
- LINE 模板消息
- LINE 快捷短语
