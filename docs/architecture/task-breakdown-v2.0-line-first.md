# Development Task Breakdown V2.0

更新时间：2026-03-08

本文档把 2.0 LINE First 升级拆成可执行开发任务，按优先级排序。

## 1. 总体策略

采用“先数据兼容，再页面切换，再清理旧逻辑”的方式推进：
1. 先新增模型与字段
2. 再落地图墙后台
3. 再切换前台首页和预约链路
4. 再打通 LINE 自动消息
5. 最后移除或弱化旧邮箱路径

## 2. P0 任务：需求冻结与设计基线

### T0-1 文档冻结
- 输出 `requirements-v2.0-line-first.md`
- 输出 `prd-v3-line-first.md`
- 输出 `data-model-v2.0-line-first.md`
- 输出当前任务拆解文档

### T0-2 技术方案冻结
- 确认顾客主身份切换方案
- 确认图墙模型与预约来源模型
- 确认旧邮箱查询下线路径

## 3. P1 任务：数据模型与迁移

### T1-1 Prisma schema 调整
- 给 `Customer` 增加 `customerType`、`firstBookedAt` 等字段
- 给 `LineUser` 增加首次欢迎推送状态字段
- 给 `Appointment` 增加 `showcaseItemId` 与来源字段
- 新增 `ShowcaseItem` 模型

### T1-2 数据迁移
- 生成 migration
- 对已有顾客按历史预约回填 `customerType`
- 为旧数据建立兼容默认值

### T1-3 数据访问层调整
- 修改顾客查找逻辑，以 LINE 为主
- 修改预约创建逻辑，允许记录图墙来源

## 4. P1 任务：LINE 顾客建档主链路

### T2-1 Webhook follow 处理
- 收到 `follow` 时自动创建/更新 `LineUser`
- 自动创建/绑定 `Customer`
- `Customer.customerType = lead`

### T2-2 首次欢迎消息
- 只在首次加好友时推送首页链接
- 使用 `APP_BASE_URL` 生成首页链接
- 避免重复发送

### T2-3 顾客档案联动
- 后台顾客列表支持显示潜在客户/正式顾客
- 顾客默认显示名使用 LINE 昵称

## 5. P1 任务：图墙后台

### T3-1 图墙管理页
- 新增 `/admin/showcase`
- 列表、创建、编辑、删除、上下架、排序

### T3-2 图墙图片与字段
- 支持上传/填写封面图 URL
- 标题、说明、价格文案、分类、关联套餐

### T3-3 图墙 API
- 新增 admin API
- 支持分类筛选、上下架过滤、排序返回

## 6. P1 任务：前台首页与预约流程切换

### T4-1 首页切换为图墙
- `/` 改为图墙首页
- 支持分类筛选
- 保持图墙排版稳定

### T4-2 图墙项详情跳转
- 点击图墙项进入预约页
- 套餐固定为该图墙项绑定套餐
- 不提供套餐切换能力

### T4-3 预约提交改造
- 顾客只提交时间和备注
- 创建预约时写入 `showcaseItemId`
- 提交首笔预约后把顾客从 `lead` 转成 `active`

## 7. P1 任务：LINE 自动通知

### T5-1 提交预约后的待确认消息
- 发送预约号
- 发送等待确认文案
- 发送预约详情公开链接

### T5-2 店长确认后的已确认消息
- 预约确认后主动发 LINE 文本消息

### T5-3 文案与双语
- 中日双语消息模板整理
- 统一 message builder

## 8. P2 任务：后台顾客与预约联动

### T6-1 顾客页升级
- 展示潜在客户 / 正式顾客状态
- 展示 LINE 关联状态
- 支持后台修改显示名

### T6-2 预约页升级
- 展示预约来源图墙项
- 展示顾客 LINE 关联情况
- 快捷跳转至 LINE 会话

### T6-3 LINE 会话页升级
- 从顾客页、预约页跳转到对应顾客会话
- 保持仅文本消息能力

## 9. P2 任务：预约详情与旧流程清理

### T7-1 公开预约详情页
- 可通过公开链接查看预约详情
- 与 LINE 消息中的链接一致

### T7-2 下线邮箱查询主路径
- 前台不再暴露邮箱查询入口
- 后台清理相关说明文案

### T7-3 兼容策略
- 若旧数据仍依赖邮箱查询，可保留临时兼容接口一段时间
- 但不再作为产品主路径

## 13. 我建议的首个开发批次

第一批次只做这 6 项：
1. Prisma schema 与 migration
2. `follow` 自动建档
3. 首次欢迎消息
4. 图墙模型与后台管理 API
5. 图墙首页
6. 从图墙进入预约并写入 `showcaseItemId`

理由：
- 这 6 项能先把 2.0 主路径跑通
- 旧系统还能继续存在，方便兼容迁移
- 后续预约详情页和后台联动都能在这个基础上稳定扩展
